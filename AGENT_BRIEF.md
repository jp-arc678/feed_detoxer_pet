## 1. Project concept (condensed)

ScreenPet is an **Android-only** screen-time companion. The user picks specific apps to monitor (e.g. YouTube, TikTok). A "pet" tracks time spent in those apps, intervenes during long sessions with escalating emotion, and reflects the user's habits over time. The pet speaks lines generated by an LLM according to a user-defined persona, animates via Rive, and behaves like an interactive desktop pet on its settings screen.

Platform is Android-only by deliberate decision (iOS does not expose per-app usage or allow overlays over other apps). **Do not add iOS-specific code paths.**

## 5. Architecture recap

Three execution contexts that **do not share memory**: main app UI, overlay (its own Flutter engine), background service.

**Layers:**
- **L0 — Brain (Kotlin foreground service):** authoritative clock; session state machine `IDLE → GUARD_PROMPT → ACTIVE(elapsed) → ENDED`; emits `sessionStarted` / `thresholdCrossed(t)` / `sessionEnded(outcome)`; debounces sub-3s flicker.
- **L1 — Detection:** poll `UsageStatsManager` (~1s), filtered to selected target apps only.
- **L2 — Orchestration:** per-app thresholds, overlay cadence, mood rules.
- **L3 — Overlay (Flutter, separate engine):** guard page + stateless pet overlay.
- **L4 — Persistence:** append-only sessions; single write-owner.
- **L5 — Presentation:** the four screens (§6.4).

## 6. Feature specifications

### 6.1 App usage access
- Read **per-target-app** foreground usage: number of opens today and duration of each session.
- Aggregate into daily totals per app for charts and history.
- Only selected target apps are ever read (privacy guarantee).

### 6.2 Target apps, per-app thresholds, per-app history
- User can **list installed apps** and add/remove them to a **target list**.
- Each target app has its **own alarming threshold** (e.g. YouTube 10 min, TikTok 5 min). The Brain activates interventions **per app, against that app's threshold**.
- Each target app has its **own screen-time history** view (separate charts/records per app).

### 6.3 Escalating alarm + AI text
- During an active session, the pet reappears periodically. **Each reappearance, expression escalates** with elapsed time relative to that app's threshold: cheerful → worried → pleading → distressed.
- The displayed line is **generated by the LLM** from the user's persona settings + live context (app, elapsed time, record). See §8.1. Always have an instant fallback line (§8.1).

### 6.4 Screens

**(a) Home Page**
- The pet appears here (idle + interactive).
- Charts of screen-time per target app (today / recent).
- Pet **greets and comments** on the user's record (LLM line, non-time-critical → can generate live with a loading shimmer).

**(b) Target List Page**
- Shows the list of target apps; add new / remove.
- Tapping an app icon → Target Detail Page.

**(c) Target Detail Page**  *(note: rename from the original "target customization page" to avoid ambiguity with the list page)*
- Set this app's alarming **threshold**.
- Show this app's **screen-time history** (opens, durations, trend).
- Optional per-app settings (intensity, enabled toggle).

**(d) Pet Settings Page**
- Persona prompt fields: **pet name**, **what the pet calls the user**, **tone**, **language**, **reminder notes**.
  - Example persona → "friendly cutie, speak english, remind about project" produces, on a 10-minute alarm: *"howdy~ it's been 10 minutes now~ your project is still waiting for you, buddy~"*.
- This screen also hosts the **interactive desktop-pet area** (touch/poke, drag; optional items/minigame later — §8.3).

## 8. Design of the unfinished concepts

### 8.1 LLM Dialogue Layer (design this as a template)

**Package & security.** Use `firebase_ai` (Firebase AI Logic). The older `google_generative_ai` and `firebase_vertexai` packages are deprecated. **Do not embed an API key in the client** — Firebase AI Logic acts as a secure proxy; enable Firebase App Check. For development, the Gemini Developer API tier (via Firebase AI Logic) is sufficient and low/no cost; start with the **Gemini 2.5 Flash** class model and keep the model name in config. Do **not** commit `firebase_options.dart` or keys to a public repo.

**Provider-agnostic interface** (so the LLM can be swapped without touching UI):

```dart
abstract class PetDialogueProvider {
  Future<String> generateLine(DialogueRequest request);
}

class DialogueRequest {
  final DialogueTrigger trigger;      // greeting | recordComment | alarm | interaction
  final PetPersona persona;           // name, nickname, tone, language, notes
  final String? appDisplayName;       // e.g. "YouTube"
  final int? elapsedMinutes;          // for alarm
  final int? thresholdMinutes;
  final double moodIntensity;         // 0.0 cheerful .. 1.0 distressed
  final UsageSummary? usage;          // today's stats for context
}
```

- **Prompt construction:** put persona + hard constraints (max ~140 chars, target language, requested tone, supportive/non-harmful, no medical claims) in the system instruction; put the situational context in the user message.
- **Latency vs the overlay's short life — important.** The overlay shows for only a few seconds, but an LLM call has latency. Design for this:
  1. Every line has an **instant local fallback** from a templated canned pool (e.g. `"{nickname}~ it's been {minutes} minutes~"`), shown immediately.
  2. **Pre-generate** upcoming alarm lines in the background **at session start** (request lines for the next thresholds) so they are cached before the overlay fires.
  3. On the Home Page (not time-critical), generate **live** with a loading shimmer.
- **Caching:** cache by `(trigger, moodBucket, app)` and rotate a few variants to cut cost and add variety.
- **Failure handling:** on any error/timeout, fall back to the canned pool. **Never block the pet on the network.**
- **Rate limiting:** debounce; cap calls per session.

Agent deliverable: the interface, a `FirebaseAiDialogueProvider` implementation behind config, a `CannedDialogueProvider` fallback, the prompt builder, and the canned templates. Make it runnable with the canned provider even before Firebase is configured.

### 8.2 Rive integration: contract + placeholder

The agent does **not** create real animation. Instead:

1. Define and document the **state-machine contract** the code depends on. The human builds the `.riv` to match these exact names:

```text
Artboard:           "Pet"
State Machine:      "PetSM"
Number input:       "mood"        // 0.0 cheerful .. 1.0 distressed   (drives expression blend)
Trigger inputs:     "greet", "poke", "celebrate", "cry", "wave"
Boolean inputs:     "isDragging", "isSleeping"
```

2. Provide a typed controller so the rest of the app never touches Rive internals:

```dart
abstract class PetController {
  void setMood(double value);     // 0..1
  void fire(String trigger);      // "poke", "greet", ...
  void setDragging(bool value);
}
```

3. Ship **two implementations**: `PlaceholderPetController` (renders a simple shape/emoji that visibly reacts to mood and triggers, so the whole app runs now) and `RivePetController` (loads `assets/rive/pet.riv`, wires `PetSM` inputs). Selecting between them is a one-line config flag, so the human swaps in their `.riv` with **zero code changes**.

4. Rive setup notes for the agent: initialise the Rive runtime before `runApp`, load `.riv` assets asynchronously (never inside `build`), and wrap the pet widget in a `RepaintBoundary`.

5. Create an empty `assets/rive/pet.riv` placeholder reference and a `PET_RIVE_CONTRACT.md` in the repo restating the contract above for the human.

### 8.3 Pet interaction on the Settings Page (recommendation)

The human is undecided. **Recommendation: split into Core and Stretch so it never blocks the build.**

- **Core (build now):**
  - **Tap = poke** → fire `poke` trigger (reaction animation) + optional short interaction line. Feeds a small positive signal to the bond system.
  - **Long-press + drag** → set `isDragging=true`, move the pet widget, settle on release.
- **Stretch (design the hook now, implement later):**
  - **Items (food/toys) + minigame** for relationship-building. Don't implement yet, but route all interactions through a single `onPetInteraction(type)` method that feeds §8.4, so adding items later needs no refactor.

Deliverable: gesture handling + the `onPetInteraction` hook wired to the bond system; items/minigame left as a documented TODO behind the hook.

### 8.4 Mood / bond system (design this)

Three timescales, all feeding the single Rive `mood` input plus triggers:

1. **Bond score (slow, persistent).** Grows on positive events, decays slowly.
   - `+` user stops within threshold (compliance), maintains a streak, positive interaction (poke/play).
   - `−` repeated overruns, long neglect (no good days).
   - **Weight positives higher than negatives** (healthy, non-guilt design).
2. **Mood baseline (medium).** Derived from recent (e.g. last 3 days) compliance + bond. Sets the pet's **resting demeanour on the Home Page** — a doing-well user meets a cheerful pet; a struggling user meets a gently concerned but still supportive pet.
3. **Session mood / intensity (fast, transient).** During an active session: `mood = clamp01(elapsedMinutes / thresholdMinutes)` blended with a small baseline offset. This is what escalates the overlay expression in real time.

Mapping:
- Overlay during session → push the **session intensity** into `mood`; fire `cry` near the high end.
- Home Page → push the **mood baseline** into `mood`; fire `greet`/`celebrate` on milestones (streak hit, improved day).

Keep all weights/decay as **named tunable constants** in one config file so the human can balance feel without hunting through logic.

## 10. Build process — phased, with Definition of Done

Run in order. Each phase must build, run on device, and be committed before the next.

**Phase 1 — Foundation**
Scaffold structure (§9), theme, navigation between the four screens (placeholder content), tunable-config file, DB initialised.
*DoD:* app launches; can navigate all four screens.

**Phase 2 — Permissions & onboarding**
Guided flow for Usage Access, draw-over-other-apps, and battery-optimization exemption, with status checks.
*DoD:* user can grant all permissions through in-app guidance; app reflects granted state.

**Phase 3 — Detection + Brain (native)**
Kotlin foreground service, `UsageStatsManager` polling, session state machine, debouncing, event emission to Flutter via channel; `BOOT_COMPLETED` + `WorkManager` watchdog.
*DoD:* on a **real device**, opening/closing a chosen app produces correct `sessionStarted/ended` events in logs.

**Phase 4 — Persistence + usage data**
Models, append-only `SessionRecord`, derived `DailyAggregate`; target-app selection + per-app threshold storage.
*DoD:* sessions persist; per-app daily totals query correctly.

**Phase 5 — Screens with real data**
Home charts, Target List add/remove, Target Detail threshold + per-app history.
*DoD:* all screens show real stored data; thresholds editable per app.

**Phase 6 — Pet placeholder + mood engine**
`PetController` + `PlaceholderPetController`; mood/bond engine (§8.4) wired to session outcomes and Home baseline.
*DoD:* placeholder pet visibly changes with mood and reacts to triggers, driven by real data.

**Phase 7 — Overlay (guard + escalating pet)**
Guard page (10s skippable), stateless pet overlay wired to Brain `thresholdCrossed` events, per-app threshold escalation.
*DoD:* on device, opening a target app shows the guard, then the placeholder pet escalates over the session.

**Phase 8 — Dialogue layer**
Canned provider first (works offline), then `firebase_ai` provider behind config; pre-generation + caching + fallback (§8.1).
*DoD:* alarm and home lines appear instantly (canned), and AI lines appear when configured.

**Phase 9 — Pet interactions (Core)**
Tap-poke + drag on Settings page; persona fields saved and fed into dialogue + Rive contract.
*DoD:* poking/dragging works; persona changes affect generated lines.

**Phase 10 — Rive swap-in & hardening**
Wire `RivePetController` to load the human's `.riv` (once provided); refine debouncing, watchdog, OEM testing, polish.
*DoD:* real `.riv` renders against the contract with no other code changes; stable across an extended session.
